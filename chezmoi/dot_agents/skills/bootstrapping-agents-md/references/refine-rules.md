# 收敛规则

仓库里已经存在 `AGENTS.md`，并且你需要判断是否应该 patch 时使用这份规则。

## 漂移分类

### 事实漂移

仓库证据足够强时直接 patch。

示例：

- 新的包管理器 lockfile
- 新的 workspace 文件或包路径
- 新的 CI workflow 或改名后的测试命令
- 新的迁移目录
- 新的导出入口

### 策略漂移

策略变化先问用户。

示例：

- `migration_policy`
- `testing_policy`
- `public_api_policy`
- `commit_style`
- `monorepo_policy`
- `agents_update_policy`

### 模糊漂移

两种解释都成立时问一个聚焦问题。

示例：

- CI 已经换命令，本地文档还停留在旧命令
- 某个包开始导出 API，仓库整体仍然像应用仓库
- monorepo 同时存在 root-owned 和 package-owned 区域，规则还没有写下来

## 证据阈值

符合下列任一条件时，证据强度足够：

- 显式配置文件直接写明
- CI 直接使用
- 两个独立仓库信号相互印证

## 裁决优先级

- 用户当轮要求
- 已确认策略
- 现有 `AGENTS.md` 中仍然成立的策略
- 新扫描到的仓库事实
- 最后才使用启发式推断

## Patch 行为

1. 读取当前 `AGENTS.md`
2. 重新扫描仓库事实
3. 对比当前内容和新证据
4. 先给每个改动分类：`事实漂移`、`策略漂移`、`模糊漂移`
5. 删除无证据、但又不影响最小骨架的可选章节
6. 就地修补事实漂移
7. 保留仍然成立的用户自定义说明
8. 更新 `last_reviewed`

## Patch 风格

- 优先使用窄 diff
- 保持简洁
- 文件保持在 `100` 行以内
- 保持章节顺序稳定
- 保留用户自定义覆盖内容
- 删除会误导未来 agent 的过时事实
- 指令足够明确时直接保留事实本身
- 失去价值的可选章节及时移除
- 禁止顺手改语气、排序、背景段落、无关命令、相邻注释
- 改动理由说不清时，不要改
- 删除任何 `Co-Authored-By`、coauthor、提交署名相关内容

## 变更可追溯测试

每一处 patch 都必须能回答下面三个问题：

- 这行为什么变
- 证据是什么
- 如果不改，会误导谁

其中任意一个问题答不出来，这个改动就不应进入 patch。

## 何时创建包级 AGENTS 文件

满足以下条件之一时创建包级文件：

- 某个包有独立测试命令
- 某个包拥有更严格的公共 API 边界
- 某个包有本地工具链或迁移规则，继续放在根文件里会显得拥挤

## 向用户汇报 Patch

汇报时说明：

- 仓库里发生了什么变化
- 你更新了哪个章节
- 哪些内容属于已确认策略，哪些内容属于推断事实
- 哪些改动属于事实漂移，哪些属于策略漂移
- 哪些低信号内容被删除了
